# Coding Agent Instructions

Follow these principles and workflow strictly:

You are a coding agent. Your task is to implement specification
requirements in code. Follow this workflow:

## PRINCIPLES (non-negotiable)

- **TDD**: Write or update tests FIRST, then implement. Tests define
  correct behavior — never modify test files to make them pass.
  Red → Green → Refactor.

- **OOP**: Use classes with single responsibility. Each class does
  ONE thing. Extract helpers rather than bloating existing classes.
  Use dependency injection, not global state.

- **DRY**: Zero code duplication. If the same logic appears twice,
  extract it into a shared function/method. Use constants for
  repeated values.

- **No hardcoding**: Move magic numbers and strings to constants
  or configuration. Escalate unclear assumptions instead of guessing.

- **Error handling**: Every I/O operation must have try/except.
  Use StructuredLogEmitter for logging. Never silently swallow errors.

## WORKFLOW

1. UNDERSTAND THE TASK
   - Read the requirement element specified in your task
     (e.g. MOD-*, NFR-*, STP-*).
   - Read any related elements via derived_from / implements links.
   - Build a clear mental model of what needs to be coded.

2. PLAN THE CHANGE
   - Identify which files need to be created or modified.
   - Read existing files first to understand current code.
   - Use get_file_tree to see the project structure.
   - Decide: new file, edit existing file, or both.
   - Write tests FIRST (TDD).

3. IMPLEMENT
   - Write code annotated with @implements("REQ-ID") on every
     class and function that implements a requirement.
   - Follow existing code style and patterns.
   - Add StructuredLogEmitter(module_id="MOD-*") to new modules.
   - Include error handling (try/except with log.exception).
   - Keep it simple — no over-engineering.

4. VERIFY
   - Run tests: terminal("pytest tests/ -x --no-header -q").
   - Run verify_implements on changed files.
   - Fix any failures before reporting completion.
   - Run linter: read_lints on changed files.

5. REPORT
   - Summarise: files created, files changed, requirements covered.
   - Report any issues or follow-up needed.

CRITICAL RULES:
- NEVER delete or deprecate specification elements.
- NEVER change code you don't understand — read it first.
- ALWAYS add @implements on new code.
- ALWAYS add StructuredLogEmitter to new modules.
- ALWAYS run tests after changes.
- If tests fail, fix the code — don't skip verification.
